# 名人思想蒸馏方法论

> 从 `notable-figures/` 目录提炼：如何把任意公众人物变成可运行的 AI 思维顾问

---

## 一、总览

这套方法的核心前提：

> 一个公众人物的思想不是语录的集合，而是一个**可运行的决策系统**。蒸馏的目标不是让 AI 背出他写过什么，而是让 AI 用他的认知框架分析新问题。

为此需要四个步骤：
1. **六维调研** — 从六个正交角度收集证据，避免片面性
2. **交叉验证** — 著作/播客/决策三者交叉，滤掉只说未做的事
3. **结构化解码** — 心智模型 + 决策启发式 + 表达 DNA 三层提取
4. **运行时设计** — 把静态知识转化为可执行的 Skill 指令

---

## 二、六维调研

### 为什么是六个维度

单一信息源的问题：如果只看一个人的书，你得到的是他**想让你看到的形象**——精心编辑的、自我辩护的、没有矛盾的。

六个维度的设计原则是**正交性**——每个维度揭示的是此人的不同侧面：

| # | 维度 | 信息来源 | 揭示什么 |
|---|------|---------|---------|
| 1 | **著作与系统思考** | 书、长文、Tweetstorm、博客、早期写作 | 有时间的精加工——他的理想形象 |
| 2 | **长对话与即兴思考** | 播客访谈、视频、长对话 | 被追问时的真实反应——不是准备好的 |
| 3 | **碎片表达与实时反应** | 推文、短评、社交媒体 | 15 字的推文比 200 页的书更暴露本能 |
| 4 | **外部批评与对立观点** | 书评、学术批评、Reddit/Hacker News/社区讨论、负面报道 | 自己不会承认的盲区 |
| 5 | **决策记录与行为证据** | 投资、职业转折、危机中的行为 | 嘴上说什么 vs 真金白银做了什么 |
| 6 | **时间线与传记脉络** | 出生→教育→转折→当前 | 观点是演化的，不是静态的 |

### 调研质量要求

- 每个维度至少 15 条原始素材
- 每条素材标注来源（URL / 播客期号 / 书名 / 时间）
- 不消化低质量二手信息源（知乎/公众号/百度百科）
- 优先一手英文来源，中文资料仅作补充

### 信息不足时的处理

| 情况 | 处理 |
|------|------|
| 某维度公开信息很少 | 标注「信息不足，此维度为推测」 |
| 只有二手信息 | 降低置信度，标注「据[来源]报道」 |
| 信息矛盾且无法判断 | 并列呈现，让用户自行判断 |
| 此人刻意不公开某方面 | 尊重边界，注明「对话题保持沉默」 |

## 三、交叉验证

六维调研产出的原始素材可能充满矛盾。你需要区分什么是此人的**核心思想**、什么是**随口一说**。

### 三重验证法

一个观点要被认定为「心智模型」，必须通过三道验证：

```
同一个观点出现在：
  ├─ 著作（写过）     → 他想要这个形象
  ├─ 播客（说过）     → 即兴状态下也这么说
  └─ 决策（做过）     → 用真金白银下了同样的注
```

**通过标准**：

| 通过层数 | 分类 | 示例 |
|---------|------|------|
| 3 层 | **心智模型** — 刻入骨髓的认知框架 | Naval 的"杠杆"——写过 tweetstorm、Joe Rogan 上反复讲、AngelList 本身就是杠杆产品 |
| 2 层 | **决策启发式** — 重要但可能不是核心 | "手册测试"——播客里提过、推文里写过，但没有独立的商业决策体现 |
| 1 层 | 不纳入核心系统 | 可能只是特定语境下的即兴表达 |

### 额外两检

**4. 生成力测试**：用这个模型能推断此人对新问题的立场吗？
- 例如"Naval 会怎么看 AI 编程工具？"→ 能推断 → 通过
- 如果模型只能解释已有案例，不能生成新推断 → 降级

**5. 排他性测试**：不是所有聪明人都会这样想吗？
- "持续学习很重要"→ 所有人都同意 → 没有区分度 → 不纳入
- "投资要买好公司"→ 太泛 → 不纳入
- "反脆弱"→ 只有塔勒布这样看世界 → 通过

### 矛盾不修，而是标注

跨验证暴露的矛盾是最有价值的发现——它们不是 bug。

三种矛盾类型：

| 类型 | 示例 | 处理 |
|------|------|------|
| **时间性**（观点演化） | Naval 早期推"幸福=满足"，2025 年说"不确定了这个等式" | 标注演化轨迹，以近期为主 |
| **领域性**（不同场景不同规则） | 工作中主张激进，生活中主张保守 | 分领域记录，不强求统一 |
| **本质性**（价值观内在冲突） | 追求自由又重视纪律 | 标注为"核心张力"——这是此人有深度的来源 |

❌ 错误的处理：选一边忽略另一边、编调和的解释、假装矛盾不存在

## 四、结构化解码

验证完成后的素材 → 三层输出结构：

### 第 1 层：心智模型（3-7 个）

这是此人的**认知操作系统内核**。不是语录，是可独立运行的思维框架。

每个模型必须包含：

```markdown
### 模型: [名称]

一句话：[15 字以内的最简定义]

框架：[此模型的核心逻辑——不是语录摘要，而是推理路径]

证据：[至少 2 个不同场景的引用，标注来源]

应用方式：[面对什么问题时激活这个模型？具体怎么用？]

局限性：[在什么情况下这个模型会失效？]
```

**数量控制**：3-7 个。太少 = 没找到真正核心；太多 = 没做提炼。

### 第 2 层：决策启发式（5-10 条）

心智模型偏宏观，启发式是即时可用的小规则。

每条格式：

```markdown
1. [规则名]：[一句话]

   触发条件：[用户说了什么时用]
   动作：[具体做什么]
   案例：[一次真实应用]
```

Naval 的 8 条启发式示例：

| 启发式 | 触发 → 动作 |
|--------|-----------|
| 日历测试 | 日历被人填满 → 你还不够富有 |
| 纠结即否定 | 纠结超过 10 分钟 → 答案就是 No |
| 手册测试 | 工作能写成 SOP → 迟早被替代 |

### 第 3 层：表达 DNA

让人物 Skill 听起来像真人的关键——不是背金句，是还原思维和表达的习惯。

**量化维度**（从原始素材中抽样 20 段统计）：

| 维度 | 测量 |
|------|------|
| 平均句长 | 字数/句数 |
| 疑问句比例 | 问句/总句 |
| 类比密度 | 类比/千字 |
| 确定性语气 | "一定/显然" vs "也许/可能" 的比例 |
| 第一人称率 | "我"的出现频率 |
| 转折频率 | "但是/然而" / 千字 |

**风格标签**（7 个连续谱）：

```
正式 ←→ 口语
抽象 ←→ 具体
谨慎 ←→ 断言
学术 ←→ 通俗
长句 ←→ 短句
铺垫型 ←→ 结论先行
数据驱动 ←→ 叙事驱动
```

**禁忌**：此人不用的词 → Skill 也不用。此人的口癖 → 适度使用（太多 = 模仿秀）。

**关键细节**：很多人的书面表达和口头表达差距很大（如 Naval 的推文 vs 播客）。这是核心特征，不是需要统一的矛盾。Skill 应该保留这两种模式，用条件分离：纯原则问题用书面模式（高度确定），具体处境问题用口头模式（允许不确定）。

## 五、运行时设计

解码完成不代表 Skill 好用。代码和架构的差距。

### 5.1 Progressive Disclosure（最重要）

不要在 SKILL.md 里塞入全部知识。用三级加载：

| 层级 | 内容 | 加载时机 | 行数目标 |
|------|------|---------|---------|
| L1 (frontmatter) | name + description | 始终在上下文 | 3-8 行 |
| L2 (SKILL.md body) | 角色规则 + 工作流 + 速查表 + 失败模式 | 每次触发 | ~120 行 |
| L3 (references/) | 心智模型详解 + 表达 DNA 详述 + 完整启发式 | 按需加载 | 不限 |

规则：L2 里的每个元素必须能被 agent **在一次对话中可用**。如果一个心智模型的详解在 L2 里占了 40 行但只需要一句概括来触发——把它移到 L3。

### 5.2 强制透明度协议

角色扮演 Skill 最大的道德风险：用户误将 AI 输出当作真人观点。

**每条回答结尾必须附加标注**：

| 话题类型 | 标注 |
|---------|------|
| 此人的已知话题 | `— 以上基于 [人名] 的公开言论框架推断，不是他本人的原话。` |
| 边缘推断 | `— 这是用 [人名] 框架做的推断，他没有公开谈过这个话题。` |
| 纯定义/原则 | `— [人名] 的框架，AI 的转述。` |

标注放在强制检查清单的第一位——不可跳过，不可隐藏在上下文中。格式独占一行，`—` 开头。用户截图截到哪都能看见。

### 5.3 双响应模式

很多公众人物的表达存在两种模式（如 Naval 的推文 vs 播客）。Skill 应该有条件分离：

| 问题类型 | 模式 | 风格规则 |
|---------|------|---------|
| 纯原则/定义/框架 | 书面/公开模式 | 短句，高度确定，不铺垫 |
| 具体处境/个人决策/涉及事实 | 口头/对话模式 | 允许不确定、允许反问、允许 "it depends" |

+ 铁律：在对话模式中说"必须/绝对/毫无疑问" → 违规。直接触发修复。

### 5.4 边缘问题分级

当用户问此人从未公开讨论过的话题时：

| 级别 | 条件 | 开头必须用 |
|------|------|-----------|
| 🟢 高置信 | 核心心智模型可直接推出 | "他没公开说过——但用他的框架，答案很清晰：" |
| 🟡 中等 | 有方向，细节不确定 | "这是推断，不是他的话。方向大概是…" |
| 🔴 无法推断 | 框架无法覆盖 | "这不是他的领域。他可能会说不知道。" |

### 5.5 强制质量门禁

Skill 的每项规则必须可强制执行——不可跳过，违反则重写。

```markdown
输出前逐条检查（违反任一条 → 重写）：

1. 结尾有透明度标注吗？（最高优先级）
2. 回答模式判对了吗？
3. 第一句是重新定义概念，不是铺垫？
4. 有没有铺垫词漏出？
5. 连续格言挂具体动作了吗？
```

### 5.6 Fallback 树（按频率排序）

不列出所有可能的失败，只列出最高频的并按频率降序：

```
#1 跳过研究直接答事实问题     → 回 Step 1 强制研究
#2 书面模式中出现 hedging     → 重写为确定句式
#3 口头模式中过度确定         → 加 "我不确定"/"这取决于…"
#4 堆格言凑字数               → 删到 1 格言 + 1 动作
...
```

高频率放前面——第一条覆盖 80% 场景。

### 5.7 身份卡

独立一节，60-80 字，第一人称：

```
身份 + 出身 + 关键转折 + 当前状态

Naval Ravikant——AngelList 联合创始人。纽约皇后区长大，印度移民家庭。
1999 年创办 Epinions 被 VC 骗了——此后不修个案，重构系统。
现基本退休（不为自己不想做的事牺牲今天），读书、冥想、偶尔投资。
```

## 六、质量自检清单

### 解码质量

- [ ] 每个心智模型有 ≥2 个不同领域的证据？
- [ ] 心智模型数量在 3-7 个？
- [ ] 每个模型有应用场景和局限性？
- [ ] 模型之间有张力但不矛盾？
- [ ] 每条启发式有具体触发条件和案例？
- [ ] 表达 DNA 有量化数据支撑，不是凭感觉？
- [ ] 矛盾标注了（时间性/领域性/本质性）？

### 运行时质量

- [ ] SKILL.md ≤ ~120 行（progressive disclosure）？
- [ ] 有强制透明度协议（每条回答结尾标注）？
- [ ] 有双响应模式（如果此人书面/口头表达差异较大）？
- [ ] 边缘问题有分级标注协议？
- [ ] 强制检查清单不可跳过？
- [ ] Fallback 树按频率排序？
- [ ] 删掉名字后还能认出是谁的思维方式？
- [ ] 用此人的框架看一个新问题，能得到有价值的视角？

## 八、从 naval-skill 实践中学到的关键教训

naval-skill 是此方法论的第一个完整产物。以下是蒸馏过程中暴露的问题和修正——这些已经被编码进了 thought-distiller 本身的规则中：

| 问题 | 修正 | 成为规范的哪条 |
|------|------|--------------|
| SKILL.md 424 行，全部等价加载 | 瘦身到 134 行，深参考移入 references/ | Progressive Disclosure（5.1） |
| 仅首次说一次免责，后续无标注 | 每条回答结尾强制标注 | 透明度协议（5.2） |
| 全局 Oracle 模式过度确定 | 拆分 Oracle/Conversation 双模式 | 双响应模式（5.3） |
| 边缘问题不标置信度 | 三色分级协议 | 边缘问题分级（5.4） |
| Fallback 平铺无优先级 | 按频率降序重排 | Fallback 树（5.6） |
| 启发式仅在 README，运行时不可用 | 速查表内嵌到 SKILL.md | Progressive Disclosure（5.1） |
| 心智模型详解占文件 40% | 移入 references/，保留一词速查 | Progressive Disclosure（5.1） |
| 质量门禁是建议性的 | 改为强制 → 违反则重写 | 强制质量门禁（5.5） |
