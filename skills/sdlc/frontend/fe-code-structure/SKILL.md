# 前端代码组织

把平铺、耦合、过大的前端业务域目录重组成可维护的深模块结构。目标不是“多建几个文件夹”，而是提升 **局部性**、降低跳转成本，并让页面编排回到清晰的组合层。

## 适用对象

- TanStack Start / React / Next.js 一类以页面级业务区为单位的前端目录
- 典型目标：`sections/workspace`、`sections/app`、`sections/dashboard`、`routes + page shell` 相关模块
- 现象信号：目录平铺、页面文件过大、浮层/文件服务/共享类型混在一起、改一个交互要来回跳多个文件

## 输出标准

- 用户行为不漂移：先重组结构，再考虑功能变化
- 分组先服务认知，再服务文件树美观
- 页面主文件只保留编排职责
- 共享协议、页面壳、浮层、文件域、侧栏域分开承载
- 新 seam 要能降低改动面，而不是增加转发层

## 工作流

## 阶段模式

**Plan 模式**：只读分析。用于形成结构地图、候选写入范围、重组风险、验证建议和并行安全判断；不得移动文件、修改 imports、创建新模块或运行会改变项目状态的重构动作。

**Implement 模式**：执行已批准计划中的结构重组。只能在批准写入范围内移动/拆分模块、调整 imports、收敛出口并运行验证；若发现需要扩大范围，按偏差协议处理。

### 1. 先建立现状地图

- 识别真实入口：路由文件、页面壳、主 hooks、对话层、共享 types/utils/api
- 记录大文件、重依赖文件、被高频引用的共享模块
- 标出“只是摆放在一起”与“本质属于同一业务子域”的文件

### 2. 按职责分组，不按技术名词分组

优先考虑这类分组：

- `shared`：types、utils、api、constants、通用状态壳
- `page`：页面编排、主 timeline、sidebar、composer、页面级 hooks
- `files`：文件树、文件浏览、文件抽屉、文件服务相关 hook
- `dialogs`：运行状态、git、skills 等浮层模块
- `rail`：右侧 rail 与其状态管理
- `list`：列表页、目录聚合页、目录转换逻辑

如果目录语义不同，可以换名字；但必须让调用者一眼看出 seam。

### 3. 做减法，识别浅模块

重点检查：

- 页面主文件是否同时承担数据拉取、浮层装配、按钮区、内容区、抽屉区
- 一个 hook 是否同时负责多个子域状态
- 一个模块是否只是把参数原样转发给另一个模块

通过 deletion test 判断：删掉这个模块后，复杂度是集中还是只是换个地方出现。

### 4. 拆页面壳，保留组合力

优先把页面主文件拆成：

- header shell
- content shell
- overlays shell
- domain hook assembly

要求：

- 主页面可以被顺着读完
- 子壳只承接清晰的一层职责
- 不为“拆而拆”；如果子模块没有独立认知价值，就不要抽

### 5. 收敛共享出口

- 对外高频依赖通过稳定出口暴露
- 同目录内部若会形成循环或测试解析问题，优先直连具体文件
- 路由层只依赖少量稳定 seam，不直接穿透业务深层结构

### 6. 验证

- 运行格式化、类型检查、构建
- 若测试失败，先判断是业务回归，还是测试 seam / alias / 测试替身脆弱
- 汇报时明确区分：结构重组完成、运行验证通过、测试基础设施待补

## 交付时必须说明

- 新分组结构及其职责
- 哪些大文件被降维成组合壳
- 哪些共享模块被收拢为稳定 seam
- 哪些验证已通过，哪些风险保留在测试层

## 禁忌

- 不要一上来改视觉或交互行为
- 不要把每个小文件都塞进 barrel，制造新循环
- 不要用“components / hooks / utils”作为唯一分组依据
- 不要为了目录整齐拆出大量没有实际收益的转发模块
- 不要忽略现有路由入口和真实调用关系
